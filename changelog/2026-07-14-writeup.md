# 2026-07-14

This week's sweep confirms last week's concern: the variation in complexity scores is systematic enough to worry about. Rule 99, which leads at 0.9667, dropped from 0.9712 on 2026-07-13. Rule 57 fell from 0.9702 to 0.9635. Rule 226 declined from 0.9384 to 0.9284. These aren't measurement noise at the third decimal place—they're consistent shifts of 0.004 to 0.01 within a day.

Worse, the ranking composition keeps shifting. Rules 31, 7, and 21 appear in this week's top six where they weren't highlighted before. Rule 184, which ranked fourth in the previous sweep, doesn't appear here. If our heuristic were stable, we'd expect roughly the same rules to cluster at the top each run, varying only in their internal order. Instead, the membership of the top tier itself is drifting.

The most likely culprit is stochastic initialization. Each rule gets seeded with a random bit string and run for a fixed number of generations. If the same rule run twice with different seeds produces geometrically different patterns, the complexity score will vary. That's not inherently bad—it means these rules are genuinely sensitive to initial conditions, which is real behavior worth measuring. But it does mean a single score per rule per week is too sparse. We're effectively sampling a distribution without reporting error bars, then treating each sample as ground truth.

We have two paths forward: either commit to deterministic, reproducible seeds—so anyone can verify that a score for rule 99 represents a specific, fixed computation—or run each rule multiple times per sweep and report mean and variance. The current setup does neither. We render galleries from single runs and publish scores that shift by 0.01 from day to day. That's not science. It's not even reliable engineering.

This isn't a surprising finding. It's exactly what the 2026-07-13 entry predicted, now confirmed. The next work isn't interpretation; it's getting the foundation solid.
